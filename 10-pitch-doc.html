02 球場

審閱者在 Acme 中留下回饋，然後將其重新輸入到電子郵件、Notion 和 edit-bay 軟體中 — 流失訪談中被提及最多的一個缺口（佔離職團隊的 41%）。annotation匯出使用我們已經運行的基礎設施，一鍵將任何選擇的annotation和繪圖轉換為 SRT、CSV 或 PDF。它的構建、標記和負載測試為每個會話 12k 個annotation，並且一次切換即可逆轉。從這裡到發貨之間唯一的事情就是下面的四個簽名。

3·反對意見

03 審稿者會問什麼

Q1這是否會將僅限內部的留言洩露給客戶？
否。匯出透過與審核頁面相同的可見性解析器運行：annotation已標記 internal 除非出口商持有，否則被排除在外 workspace:admin 並透過標記的複選框明確選擇加入。涵蓋 14 項許可矩陣測試。
↗ 規範 §4.2 — 可見性與權限
Q2一個巨大的會話會發生什麼事——數千個annotation？
匯出按每批 500 個annotation進行分頁並串流傳輸檔案。針對prod 中最大的真實會話（12,400 個annotation的機構審核）的壓力運行在 3.1 秒內完成；典型會話的 p95 為 340 毫秒。
↗ 實作筆記，06-24 — 負載測試結果
第三季為什麼使用文件格式而不是公共 API？
SRT + CSV + PDF 涵蓋了 212 個標記的認同票證中的 83% export-request 過去兩個季度。 API 是正確的第二階段——它需要驗證範圍工作，而這不需要。檔案現在可以解鎖使用者，而不會排除 API 設計。
↗ metric — 認同標籤分析，Q1–Q2
第四季這是否會增加我們必須營運的新基礎設施？
零新服務。 SRT/CSV 在現有工作池中內嵌渲染（services/render-workers）； PDF 重複使用已產生每週摘要的報告服務。沒有新佇列，沒有新資料存儲，沒有架構遷移。
↗ 規範 §5.1 — 架構
Q5匯出的客戶內容的合規情況如何？
每次匯出都會在文件元資料中標記workspace ID、請求使用者和時間戳，並寫入審核日誌 (audit.annotation_export）。文件透過過期的簽名 URL（24 小時）提供，並且不會在該視窗之外儲存。
↗ 規範 §6 — 審核與保留

5·風險+需求

05 風險、回滾以及我需要你做什麼

風險與回滾

  • 回滾是一種切換。 翻轉 annotation_export off — 沒有交付模式遷移，沒有什麼可以放鬆的。
  • 爆炸半徑是有限的。 匯出在共用工作池中執行，每個作業逾時 30 秒；失控的作業不能導致審核回放不足。
  • 最糟糕的可信失敗： 如果簽名 URL 清理滯後，則孤立的匯出檔案 — 無論如何，它們都會在 24 小時內自行過期。
  • 已知差距： 繪圖annotation扁平化為 CSV 中的邊界框（完整向量僅限 PDF）。產品內記錄；第 2 階段項目。

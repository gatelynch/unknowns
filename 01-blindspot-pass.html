未知中的未知：Acme auth 模組

已掃描 services/auth/ （47 個文件），自 2025 年 1 月以來有 214 次提交涉及它，3 次遷移正在進行中，2 次恢復嘗試。你會透過艱難的方式學到的七件事。

你要求什麼

「新增的 SSO provider」 — 聽起來像是：實作一個接口，註冊它，寫redirect處理程序，完成。在乾淨的程式碼庫上，這是一個為期兩天的任務。

你實際上正在走進什麼

一個模組 會話儲存的遷移中期, 與 一個悄悄繞過middleware的provider 每個人都假設跑步， 恢復之前的嘗試 正是在這個任務上，並且 身份關聯模型 從provider介面中根本看不到這一點。

盲點
地雷 01

會話是雙重寫入的——你首先發現的 Redis 儲存是錯誤的

Acme 正在從 Postgres 認同的會話遷移到 Redis (MIG-118，自三月以來陷入停滯）。目前每個會話寫入都會經過 SessionBridge.write()，扇出到 兩者 RedisSessionStore 和遺產 pg_sessions 表。遷移從未完成，因此讀取仍然來自 Postgres。

為什麼會咬人尋找“會話”範圍 RedisSessionStore 首先——它看起來很規範，有最新的程式碼和最好的文件。直接寫入它，你的登入將在 dev 中正常工作（在那裡啟用了 Redis 讀取），而在 prod 中則默默地失敗，其中讀取仍然會命中 Postgres。
當你為新provider建立會話時，請透過 SessionBridge 路由所有內容 - 切勿直接透過 RedisSessionStore 或 pg_sessions 進行路由。在編寫任何會話程式碼之前解釋一下 MIG-118 讀/寫分割。
地雷 02

SAML provider（你顯而易見的範本）繞過了驗證middleware

除 SAML 之外的所有provider均安裝在 authPipeline，它運行速率限制、審核日誌記錄和設備信任檢查。 saml/handler.ts 將其 ACS 端點直接掛載在路由器上，因為 SP 發起的 POST 回呼在 2024 年破壞了 CSRF 層，並且沒有人回來。有一個 // TODO(priya): fold back into pipeline 從 14 個月前開始。

為什麼會咬人SAML 是最近新增的provider，因此它是自然要複製的檔案。複製它，你的新provider就沒有速率限制，也沒有audit trail——這不會導致任何測試失敗，因為合約測試不會斷言middleware的存在。
使用 oauth/google.ts 作為結構模板，而不是 saml/handler.ts — SAML 故意繞過 authPipeline。確認新provider的路由已安裝在 authPipeline 內並顯示安裝點。
歷史 03

有人已經嘗試過這個——並且由於一個仍然有效的原因而被恢復

PR #2841（“新增 Okta OIDC provider”，2025 年 9 月）已合併並在 9 天後恢復。恢復提交解釋了原因：Acme 回呼 URL 必須攜帶租使用者後綴 (/auth/callback/:provider/:workspaceSlug）因此workspace範圍的邀請連結在重定嚮往返後仍然存在。 Okta 嚴格的redirect URI 匹配拒絕了動態後綴，並邀請使用者登入錯誤的workspace。每個人都同意的修復方案 - 簽署的狀態參數workspace聲明 AUTH_STATE_SECRET - 是在恢復線程中設計的，但從未構建。

為什麼會咬人當前程式碼中沒有任何暗示這一點。你將重新實作 #2841 的方法，透過每個本機測試（單一workspace開發環境），並在被邀請的使用者第一次登入時重新發現生產中的錯誤。
首先閱讀 PR #2841 的回應。實作它建議的簽章狀態參數workspace聲明，而不是帶有租使用者後綴的回呼 URL，並為workspace範圍的邀請流程新增測試。
缺少概念 04

身分和帳戶是不同的東西——provider介面隱藏了它

極致劈叉 users 來自 identities：一個帳戶可以有多個連結的身份（電子郵件+密碼、Google、SAML），加入到 identities 表。provider介面僅傳回一個設定檔；連結決策發生在 identity/linker.ts，輸入經過驗證的電子郵件。無法保證電子郵件經過驗證（許多 SSO IdP 不保證）的provider必須設置 requiresManualLink: true 或者連結器會很樂意合併兩個共享公司別名的不同人。

為什麼會咬人天真地實作介面，一切正常 - 直到客戶的 IdP 發送未經驗證或重新分配的電子郵件，並且你的provider默默地將陌生人的身份附加到現有帳戶。這是帳戶接管錯誤，而不是登入錯誤。
在實作之前引導我了解一下 Identity/linker.ts。明確決定此 IdP 是否保證經過驗證的電子郵件；如果沒有，則設定requiresManualLink並建立手動連結確認路徑。
地雷 05

令牌刷新位於一個標誌後面，該標誌在開發中打開，在生產中關閉

刷新令牌輪換由以下因素控制 auth.refresh_rotation 在 flags.yaml。它預設為 true 在開發和分階段， false 在prod 中（五月時鐘偏離事件後暫停推出）。隨著旗幟的關閉，產品會議將準確地進行 ACCESS_TOKEN_TTL (12h)，然後硬過期－沒有刷新路徑。

為什麼會咬人你編寫的任何刷新邏輯都將針對開發行為和prod 中的死程式碼進行測試。更糟的是：如果你的provider假設發生了刷新，那麼產品使用者每 12 小時就會在會話中登出，並且錯誤報告會指出“SSO 隨機將我登出”，並指向你的程式碼。
針對 auth.refresh_rotation=false 實作（產品現實）。讓provider在 12 小時硬性期限內工作，並單獨注意如果恢復輪換推出會發生什麼變化。
公約 06

註冊provider需要三個步驟，並且只有兩個步驟是可發現的

provider在編譯時並不活躍。它必須 (1) 註冊於 providers/registry.ts，(2) 插入 auth_providers 透過遷移表 - 管理 UI 讀取資料庫，而不是註冊表 - 並 (3) 新增到固定列表中 __tests__/provider-contract.規格說明.ts，它針對每個註冊的provider運行共享合約套件（狀態驗證、重播保護、登出扇出）。如果登錄機碼和裝置清單不一致，CI 就會失敗。

為什麼會咬人跳過遷移，provider會編譯、通過測試，並且永遠不會出現在任何workspace的 SSO 設定中。兩位工程師為此獨立損失了半天；它不在任何人的文件中。
完成所有三個註冊步驟：registry.ts、auth_providers 遷移和provider合約固定清單。在我們完成之前，向我展示傳遞給新provider的合約套件。
地雷 07

登出不是一個函數呼叫－它是一個事件，審核player會監聽

會話撤銷發布 auth.session.revoked 在內部事件總線上。即時審閱播放器擁有自己的長期播放授權，並且僅在消耗該事件時才釋放它們。 logout.ts 為現有provider處理此問題，但 SSO 單點登出 (SLO) 回調到達單獨的路徑，並且必須自行發布事件。

為什麼會咬人錯過事件並登出「有效」——會話行消失了——但任何擁有打開的審閱選項卡的人都可以保留對私人視訊的完全播放存取權限，直到授權 TTL 到期（最長 24 小時）。對於向工作室出售審核工作流程的公司來說，這是安全事件，而不是錯誤。
處理 SLO：新provider中的任何撤銷路徑都必須發佈 auth.session.revoked。新增一個測試，斷言審核player的授權在 IdP 發起的登出時釋放。

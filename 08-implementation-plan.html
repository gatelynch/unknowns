高調整可能性

我在三個方面做出了你可能不同意的判斷。每個標記的選擇都顯示了我考慮的替代方案 - 切換以進行比較。

A1·資料模型－新 annotation_exports 表

annotation_exports 新
編號uuid PK
審核_iduuid → 留言
requested_byuuid → 使用者
狀態枚舉（排隊、渲染、就緒、失敗）
選項jsonb
快照①jsonb
① 無快照列 — 渲染器 JOIN annotation 即時渲染時
產物_url ②文字為空

② 無儲存神器 — GET /exports/:id/下載 按需渲染
created_at時間戳
選擇① 快照與即時加入
計畫的選擇——非規範化快照
將annotation有效負載複製到 快照 jsonb 當請求匯出時。匯出的是審核記錄 在那一刻：以後的編輯、解決和刪除不會重寫歷史記錄。
  • 即使審核已存檔，匯出仍然有效。
  • 渲染器讀取一行 — 沒有 N+1 跨行 annotation + annotation_replies.

Cost: 嚴格審核時約為 40 KB/行；相對於活動線程，快照可能會變得陳舊。

替代方案 - 在渲染時即時加入
跳過快照；渲染器加入 annotation 當文件生成時。出口始終反映當前狀態。
  • 沒有重複資料，沒有陳舊問題。
  • 重新下載相同的匯出可以產生 不同的 文件－對於審計用途來說令人驚訝。

若符合以下條件，請選擇此選項： 出口是工作文件，而不是記錄。一行告訴我： “使用即時加入。”

選擇② 儲存的產物與按需渲染
計畫的選擇 - 渲染一次，儲存在 blob 儲存中
工作人員渲染 PDF/CSV 一次並寫入 產物_url。下載是一個簽名 URL redirect — 便宜、可緩存，並且可以與缺乏 Acme 帳戶的審閱者共享。

Cost: 要管理的 blob 生命週期；我會新增 30 天的 TTL 掃描（C 部分，第 6 項）。

替代方案－在每次下載時渲染
沒有儲存檔案： GET /exports/:id/下載 串流新鮮渲染。零儲存空間，但每次下載大型留言約 2-4 秒，並且沒有可共享的公共連結。

若符合以下條件，請選擇此選項： 儲存/合規性規則使儲存的副本變得尷尬。

A2·新型介面

// src/types/exports.ts（新）
匯出介面 出口請求 {
審核Id: 留言ID;
format: 'pdf' | 'csv';1
range: ‘全部’ | “未解決” | 時間碼範圍;2
includeReplies: 布林值; // 預設為真
includeDrawings: 布林值; // 預設為假3
}
 
匯出介面 annotation快照 {
id: annotationID;
timecode: 字串; // "00:04:12.08"
作者：{ ID： 使用者ID;姓名： 字串 };
body: 字串;
state: ‘打開’ | ‘解決’;4
replies: 回覆快照[];
}
1從 PDF + CSV 開始。新增 'xlsx' 或 'srt' 後來是一名聯合成員+一名渲染器。
2TimecodeRange 讓製作人只匯出一個場景的annotation。如果沒有人要求這樣做，則將其剪掉可節省約 ½ 天的時間。
3繪圖（框架標記）光柵化為 PDF。預設情況下關閉，因為它會使渲染時間增加三倍。
4故意將內部 5 狀態機降為匯出中的 2 狀態。告訴我審稿者是否需要查看 'wont_fix'.

A3·使用者體驗流程

步驟1
匯出按鈕
審閱工具列溢位選單中的新項目，位於「複製共用連結」旁。
步驟2
選項模態
格式、範圍、包含回應、包含繪圖。重複使用 模態形式 從設定。
步驟3
該計畫最薄弱的部分
後台工作+吐司
在工作人員中匯出渲染；使用者會收到「我們會通知你」的提示，然後可以離開。
步驟4
就緒通知
響鈴通知+下載連結；也列在檢視 → 匯出標籤下。

為什麼第 3 步很弱： 大多数导出将在 3 秒内完成，因此“一劳永逸”可能会让人觉得有点大材小用，但包含 400 个注释的绘图审核需要大约 20 秒，这太长了，无法阻止。混合方式（最多等待 4 秒，然後回退通知）效果更好，並且會增加大約半天的時間。你的電話。

執行順序

我實際建構它的順序。每一步在 CI 上都是綠色的；在第 5 步翻轉標誌之前，不會傳送任何使用者可見的內容。

  1. 1遷移+模型。 創建 annotation_exports，連接 Sequelize 模型、工廠、夾具。0.5d
  2. 2類型+請求驗證。 A2 中的所有內容，以及 zod 模式 POST /api/審核s/:id/exports.0.25d
  3. 3Renderers. 首先是 CSV（簡單），然後透過現有的 PDF @acme/render service.1天
  4. 4Worker 作業 + blob 上傳。 排隊開啟 exports.render;然後重試×3 狀態=失敗.0.25d
  5. 5UI：選單項目、模式、Toast、匯出標籤。 旗幟後面 export_annotations.0.5d
  6. 6向 Acme 團隊進行測試、文件、標記展示。0.25d

機械工作（相信我）

調整可能性低

重構並處理功能需求。這裡沒有任何判斷力——故意倒塌。

無聊但必要 8 項任務 · 總計約 0.5 天，併入上述估算中
  • 萃取物 AnnotationSerializer 來自 api/annotations.ts 進入 lib/serializers/ 這樣出口工人就可以重複使用它。 純粹的舉動，沒有行為改變。
  • 完成 CommentMarker → AnnotationMarker 重新命名 — 3 個檔案仍匯入已棄用的別名。
  • 新增 annotation_exports 到夾具載入器和 CI 資料庫重置腳本。
  • 註冊 exports.render 在 workers/index.ts 並將其新增到死信警報列表中。
  • 創建功能標誌 export_annotations 在 flags.yaml，預設關閉。
  • 新增每晚對過期匯出 blob 進行 TTL 掃描 jobs/cleanup.ts （僅當選項 ② 保持原樣時）。
  • 延伸 openapi.yaml 使用兩個新端點並重新產生客戶端。
  • 將時間碼格式化助手從 player/utils.ts 到 lib/time.ts — PDF 渲染器需要它，並且不應從播放器包中匯入。

偏離 2（共 4）15:22

佇列往返 Date 作為字串的對象

計畫說了什麼
ExportJobPayload.requestedAt: Date，傳遞給工人並與留言進行比較 updated_at 檢測過時的匯出。
程式碼揭示了什麼
BullMQ JSON 將有效負載序列化到 Redis 中，因此worker收到 ISO 字串，而不是 Date。過時性比較默默地強制並在單元測試中通過，但在整合運行中的重新交付作業中失敗。
保守的選擇
以有線格式輸入有效負載（requestedAt: string ISO-8601)，在工作邊界處解析一次 exportConsumer.ts。沒有自訂序列化器。
重訪
其他 media-tasks 有效負載可能具有相同的潛在錯誤—作業有效負載的共用 zod 編解碼器將解決此類問題。

Zip 串流媒體已經存在——放棄了計畫 archiver 依賴性

計畫說了什麼
新增 archiver 將匯出套件（視訊 + CSV + 縮圖）建置為 S3 的 zip 串流。
程式碼揭示了什麼
packages/media/src/zipStream.ts 已經實作了具有背壓處理的串流 zip——它為批量資源下載功能提供了支持，並且自 3 月以來一直在生產。
保守的選擇
重複使用 zipStream.ts 按原樣。減少了一種依賴性，捆綁包匯出繼承了與批量下載相同的經過測試的反壓行為。
重訪
zipStream 住在 packages/media 但現在被兩個功能使用——值得推廣 packages/shared 在清理過程中。

來賓審閱者可以查看留言，但無法下載資產

計畫說了什麼
匯出權限=留言級ACL：任何可以開啟留言的人都可以匯出它。
程式碼揭示了什麼
workspace級別 guest_審核er 角色授予審閱權限 can_download_assets = false — 客戶使用它讓外部利害關係人發表留言，而無需將媒體帶離現場。僅審核 ACL 的檢查可以讓訪客透過匯出來洩露來源影片。
保守的選擇
出口退貨 403 為了 guest_審核er 跨所有格式，鏡像現有的資產下載簽入 assetAccess.ts。最嚴格的辯護閱讀。
重訪
可以說，客人仍然應該獲得 CSV（僅annotation，無媒體）。這是一個產品調用，不是我的——記錄在下面。

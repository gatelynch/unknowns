來源 provider/速率限制器（Rust，612 loc，3 個模組） 目標 acme-web 中的 src/api/retry/ 讀 Bucket.rs · backoff.rs · Budget.rs · 14 個測試
── 1. 它的作用 ──
1 Rust crate實際上做了什麼
  • 令牌桶准入 (bucket.rs)：每個傳出請求都會從重新填充的儲存桶中提取 1 個令牌 rate_per_sec = 8 最多 burst_capacity = 40。桶開始 滿，所以一個冷客戶端可以瞬間突發40個請求。
  • 填充是惰性的並且整數被截斷：只有當請求到達時才鑄造令牌，計算公式為經過的奈秒 × 速率 ÷ 10⁹。分數代幣永遠不會被儲存——相反， last_refill 只有當至少鑄造一整枚代幣時才會前進，因此子代幣經過的時間會向前推進。
  • 裝飾相關的抖動退避 (backoff.rs): 在可重試失敗時，下一個延遲統一從 [base, min(prev × 3, cap)] 與 base = 250ms, cap = 30s。每次抽籤都會為下一次抽籤埋下種子——延遲會徘徊而不是同步加倍，這會在中斷後使客戶端失去同步。
  • 重試預算 (budget.rs)：第二個單獨的桶子。每個 首次嘗試成功 要求存款1；每個 重試嘗試 提款 10. 當餘額不足以支付提款時，重試將被完全拒絕（首次請求仍會失敗）。這會將持續中斷期間的重試放大上限限制在 10% 左右。
  • 故障分類是呼叫者的工作：Rust crate從不檢查錯誤。它暴露了 acquire(), next_delay(), try_withdraw() 呼叫者決定什麼算是可重試。 TS 移植保留了這一邊界 — Acme 的 isRetryable() 在 src/api/errors.ts 仍然是唯一的事實來源。

惰性填充－整數截斷必須在浮點世界中生存

vendor/rate-limiter/src/bucket.rs:41鐵鏽
fn refill(&mut self, now: Instant) {
  let elapsed = now
    .saturating_duration_since1(self.last_refill);
  let new_tokens = elapsed.as_nanos() as u64
    * self.rate_per_sec as u64
    / 1_000_000_0002;
  if new_tokens > 03 {
    self.tokens = (self.tokens + new_tokens)
      .min(self.burst_capacity);
    self.last_refill = now;
  }
}
src/api/retry/tokenBucket.ts（建議）TS
private refill(now: number): void {
 // now comes from performance.now(), not Date.now()
 const elapsedMs = Math.max(0, now - this.lastRefill)1;
 const newTokens = Math.floor(
  (elapsedMs * this.ratePerSec) / 1000
 )2;
 if (newTokens > 03) {
  this.tokens = Math.min(
   this.tokens + newTokens,
   this.burstCapacity
  );
  this.lastRefill = now;
 }
}
1時鐘倒退。 saturating_duration_since 將負經過時間箝位為零。 Date.now() 在NTP校正下可以向後跳轉；移植使用單調 performance.now() 和 保持 Math.max(0, …) 作為腰帶和吊帶的搭配。
2整數與浮點數學。 魯斯特的 u64 除法截斷； JS 部門沒有。 Math.floor 恢復截斷。米利斯代替納米是安全的：在 rate = 8, elapsedMs * 8 遠低於 2⁵³，所以沒有精度損失。
3承重防護罩。 last_refill 只有在鑄造整個代幣時才會進步。放棄這個保護（一個簡單的「簡化」）會默默地丟棄每次調用的子令牌進度——在低速率下，桶將 從不 在頻繁輪詢下重新填入。完全保留，加上回歸測試。

fn next_delay(&mut self) -> Duration {
  let hi = (self.prev_delay_ms.saturating_mul(3)4)
    .min(self.cap_ms);
  let lo = self.base_ms;
  let ms = self.rng
    .gen_range(lo..=hi.max(lo))5;
  self.prev_delay_ms = ms;6

src/api/retry/backoff.ts（建議）TS
nextDelay(): number {
 const hi = Math.min(this.prevDelayMs * 34, this.capMs);
 const lo = this.baseMs;
 const span = Math.max(hi, lo) - lo;
 const ms = lo + Math.floor(
  this.random() * (span + 1)
 )5;
 this.prevDelayMs = ms;6
 return ms;
}
4飽和倍增。 saturating_mul(3) 守衛 u64 溢出。在 JS 中這不能溢出 - capMs = 30_000 很久以前就限制了產品 2⁵³ ——所以故意放棄了警衛（參見§3，「放棄」一欄）。
5包含與排除範圍。 魯斯特的 lo..=hi 包括兩個端點。天真 lo + random() * (hi - lo) 永不屈服 hi。的 + 1 裡面 Math.floor 恢復包容性——一個單字元的蟲子磁鐵，被召喚出來，這樣你就可以否決或祝福它。
6有狀態的種子。 每次抽籤都會成為下次迭代的抽籤結果 prev ——這就是它的原因 裝飾相關的 而不是簡單的帶有抖動的expo-backoff-with-jitter。 reset() 恢復 prevDelayMs = baseMs 成功後，匹配箱子的 Backoff::reset. this.random 可注入用於確定性測試（Rust crate使用種子 SmallRng 在其測試中）。

重試預算－事件循環不需要的互斥體（有陷阱）

vendor/rate-limiter/src/budget.rs:58鐵鏽
pub fn try_withdraw(&self) -> bool {
  let mut b = self.inner.lock().unwrap()7;
  b.deposit_drip(Instant::now());
  if b.balance >= WITHDRAW_COST {
    b.balance -= WITHDRAW_COST;8
    true
  } else {
    false // refuse retry, don’t queue9
  }
}
src/api/retry/budget.ts（建議）TS
tryWithdraw(): boolean {
 // no lock: single-threaded event loop7 —
 // but NO await between check and debit.
 this.depositDrip(this.clock());
 if (this.balance >= WITHDRAW_COST) {
  this.balance -= WITHDRAW_COST;8
  return true;
 }
 return false9;
}
7線程安全→事件循環。 Rust crate的 Mutex 存在是因為 Rust 呼叫者從工作執行緒中退出。 Acme 的客戶端在一個事件循環上運行，因此鎖消失了 — 但它所提供的原子性必須按照約定保留: tryWithdraw 是端對端同步的，我將新增一個 eslint no-await-in-budget 邊界annotation加上該方法永遠不會傳回 Promise 的測試。
8先支票後借記保持融合。 兩面形狀相同。這裡的故障模式不是資料競爭——而是未來的重構，插入一個 await （例如，記錄）在支票和借記之間，讓兩次飛行中的重試都通過檢查。
9拒絕，絕不排隊。 在預算空的情況下，Rust crate返回 false 立即 - 請求要么作為第一次嘗試發出，要么快速失敗。港口必須 不 「有幫助」將重試排入佇列以供稍後使用；這將重建預算所要預防的重試風暴。
── 3. 保留/更改/刪除 ──
3 行為：保留/故意改變/丟棄
保存良好
  • 補充截斷+防護 — 僅整個代幣； lastRefill 僅在薄荷上進步
  • 抖動公式 — 制服結束 [base, min(prev×3, cap)], 包括兩端
  • 預算經濟學 — 每次首次嘗試成功 +1，每次重試 -10，上限 1000
  • 桶子開始滿 — 40 次請求冷爆是故意的（與Rust crate測試相符） burst_at_t0)
  • 呼叫者對錯誤進行分類 — 限制器從不檢查故障
刻意改變
  • Instant → performance.now() ——都是單調的；毫秒解析度在 8/s 速率下就足夠了
  • u64 奈米 → number 女士 — 所有產品均可證明 < 2⁵³； Math.floor 重演整數除法
  • Mutex<Budget> → 平原田野 — 透過僅同步約定+測試實作原子性
  • SmallRng → 注入 random() — 預設為 Math.random, 測試中的種子存根
已刪除（不需要）
  • saturating_mul 溢出防護裝置 — 一旦在 ms 範圍內應用上限，則無法到達
  • Send + Sync 意味著， Arc 複製 — 沒有可以共享的線程
  • tokio/async-std 特徵標誌 — 移植的建構與執行時間無關
  • 普羅米修斯計數器 — Acme 透過發出 telemetry.track() 相反；保留鉤點

預算耗盡
持續 60 秒 100% 失敗		 重試約 10 次後，餘額會耗盡至 0；進一步的重試被拒絕，首次嘗試繼續。重試率確定為存款÷成本=成功率的10%（此處為0）。 相同的經濟，相同的解決點。區別：拒絕面為 RetryBudgetExhausted 錯誤，以便 Acme 的上傳佇列可以顯示“正在重新連接”，而不是默默地失敗。 同等*

速率 8/s，但每 20 毫秒輪詢一次（0.16 個令牌/輪詢）		 Guard將子令牌時間向前推進；無論投票節奏如何，每約 125 毫秒就會鑄造一次代幣。 同樣，透過保存的 newTokens > 0 守衛（A 對，註 3）。回歸測試斷言薄荷節奏為 20 毫秒輪詢。 相同的
延遲上限
連續10次失敗		 繪製範圍向 [250ms, 30s];延遲保持 ≤ 30 秒，但仍然抖動 — 從來不是固定的 30 秒（避免重新同步客戶端）。 相同的邊界，相同的非退化抖動－透過種子 RNG 重播Rust crate進行驗證 cap_still_jitters 測試向量。 相同的
